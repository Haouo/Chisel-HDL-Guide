# Part 6 — 處理器設計（單週期 → pipeline → 微架構資料結構）

對應：書 §14、§15 ｜ `import chisel3._`、`import chisel3.util._`

> 把 Part 1–5 全組起來：control（FSM）+ datapath（ALU/RF/Mem）+ 流控。
> 路線：單週期 Leros → 3-stage RISC-V（Wildcat）→ 微架構資料結構（朝 OoO 鋪路）。

本章不是要一次教完處理器設計，而是把前面所有 Chisel 技巧放進一個會跨 cycle、會有 hazard、
會需要 trace 與測試的設計中。處理器是最好的綜合題：每一條線都有時序位置，每一個 control bit
都要跟著 instruction 在 pipeline 中移動，每一個錯誤都可能在幾拍後才表現成錯誤結果。

建議學習順序是先接受限制，再逐步解除限制：先做單週期或無 hazard pipeline，確認 decode/ALU/RF 正確；
再加入 data hazard 測試，看見錯誤；接著加 stall 或 forwarding；最後才處理 branch flush、exception、
commit trace 與 difftest。不要在還沒測到第一個相依指令錯誤前就把 forwarding 寫滿。

**本章路線圖**

- 先學：單週期資料路徑、3-stage pipeline、hazard 反例、MicroOp、flush、commit/perf counter。
- 會踩坑：pipeline stage 邊界不清、相依指令讀舊值、bubble/NOP 混淆、trace 比到 micro state。
- 最後能做：用最小程式重現 CPU bug，透過 stall/forwarding/trace 把 pipeline 修到可驗證。

## M6.1 簡單處理器 — Leros（單週期 / FSMD）

單週期或 FSMD 處理器的價值是讓資料路徑完整可見：PC 如何更新、instruction 如何解碼、
register/accumulator 何時寫回、控制邏輯如何選 ALU operation。這一版可以刻意簡化 memory latency，
因為目標是先建立「一條指令從取指到寫回」的完整故事。

| datapath 元件 | 用什麼做 | 章節 |
|---------------|----------|------|
| PC | `RegInit` + 加法 | M1.5 |
| Instruction Memory | `SyncReadMem` / ROM | M2.5、Part 7 |
| Register File | `Reg(Vec)` | M1.6 |
| ALU | `switch(op)` | M2.1 |
| Control | FSM / FSMD | M3 |

```scala
class Leros extends Module {
  val io  = IO(new Bundle { val dout = Output(UInt(32.W)) })
  val pc  = RegInit(0.U(16.W)); val acc = RegInit(0.U(32.W))
  val prog = VecInit(Array(0.U(32.W) /* 機器碼，或讀檔生成 */))
  val instr  = prog(pc)
  val opcode = instr(31, 24); val imm = instr(15, 0)
  switch(opcode) {
    is(0x08.U){ acc := imm }         // load imm
    is(0x09.U){ acc := acc + imm }   // addi
    is(0x0a.U){ acc := acc - imm }   // subi
  }
  pc := pc + 1.U
  io.dout := acc
}
```
⚠️ **陷阱**：`SyncReadMem` 讀有一拍延遲；單週期教學版可用組合讀 `Mem` 簡化時序。

對應：§14。

## M6.2 RISC-V 3-stage Pipeline — Wildcat

Pipeline 的第一個版本應該故意簡單。先假設沒有 data hazard、沒有 branch，讓學生看懂 stage boundary
與 pipeline register。接著用相依指令打破假設，例如 `addi x1, x0, 1; add x2, x1, x1`，
觀察 decode stage 讀到舊值。這樣 forwarding/stall 就不是抽象名詞，而是修一個具體錯誤。

```
[1] Fetch            取指、PC+4
[2] Decode+RegRead   解碼、讀 register file
[3] Execute+Mem      ALU、記憶體、寫回
```

沒有 hazard 處理時，相依指令的錯誤來自 stage 時間點不重合：

```text
cycle       0        1        2        3        4
inst 0      IF       ID/RF    EX/WB
inst 1               IF       ID/RF    EX/WB
                         ^ reads x1 before inst 0 writes it
```

若 `inst 0 = addi x1,x0,1`，`inst 1 = add x2,x1,x1`，那 `inst 1` 在 cycle 2 讀 RF，
但 `inst 0` 的結果同一拍才在 EX/WB 產生。這就是 forwarding 或 stall 要解的時間差。

```scala
class DecodeExec extends Bundle {     // pipeline register payload（不帶方向）
  val rd = UInt(5.W); val rs1Val, rs2Val, imm = UInt(32.W)
  val aluOp = UInt(4.W); val regWrite = Bool()
}

class Wildcat extends Module {
  val io = IO(new Bundle { val dbg = Output(UInt(32.W)) })
  // Stage 1: Fetch
  val pc = RegInit(0.U(32.W)); val imem = SyncReadMem(1024, UInt(32.W))
  val instr = imem.read(pc >> 2); pc := pc + 4.U
  // Stage 2: Decode + RegRead
  val rf = Reg(Vec(32, UInt(32.W)))
  val de = Wire(new DecodeExec)
  de.rd := instr(11,7); de.rs1Val := rf(instr(19,15)); de.rs2Val := rf(instr(24,20))
  de.aluOp := instr(14,12); de.imm := instr(31,20); de.regWrite := true.B
  val deReg = RegNext(de)            // ★ pipeline register
  // Stage 3: Execute + Writeback
  val aluY = WireDefault(0.U(32.W))
  switch(deReg.aluOp){ is(0.U){ aluY := deReg.rs1Val + deReg.rs2Val }; is(7.U){ aluY := deReg.rs1Val & deReg.rs2Val } }
  when(deReg.regWrite && deReg.rd =/= 0.U){ rf(deReg.rd) := aluY }   // x0 恆 0
  io.dbg := aluY
}
```

| 概念 | 處理 |
|------|------|
| stage 隔離 | stage 間放 `RegNext(bundle)` / `Valid[T]` |
| `x0` 恆 0 | 寫回時 `rd =/= 0.U` 才寫 |
| imm sign-extend | `Cat(Fill(20, inst(31)), inst(31,20))` |
| **data hazard** | 後級結果未寫回 → forwarding / stall |
| **control hazard** | branch 改 PC → flush（插 bubble） |

⚠️ **陷阱（pipeline 必踩）**：data hazard（相依指令讀到舊值）、branch hazard（分支成立要 flush）。
教學版可先「假設無相依」，進階版加 forwarding/stall。

### Walkthrough：從錯誤程式推到 forwarding

不要一開始就把 forwarding network 寫滿。比較好的教學流程是先讓 pipeline 在一個最小反例上失敗，
再用波形與 commit trace 確認錯在哪一拍。

測試程式：
```asm
addi x1, x0, 1    # x1 = 1
add  x2, x1, x1   # 期望 x2 = 2
add  x3, x2, x1   # 期望 x3 = 3
```

在沒有 forwarding/stall 的 3-stage pipeline 中，第二條指令進 Decode+RegRead 時，第一條指令的結果還沒寫回 RF。
因此 `add x2, x1, x1` 讀到舊的 `x1=0`，最後 commit 可能變成：

```text
cycle  pc    inst             rd  rdData
  3    0x00  addi x1,x0,1     x1  1
  4    0x04  add  x2,x1,x1    x2  0   # 錯：decode 時讀到舊 x1
  5    0x08  add  x3,x2,x1    x3  1   # 錯：錯誤繼續傳遞
```

第一個修法是 stall：當 Decode 階段使用的 `rs1/rs2` 命中 Execute 階段尚未寫回的 `rd`，就凍住 Fetch/Decode，
並在 Execute 插 bubble。這最容易理解，也最容易驗證，但 CPI 會變差。

```scala
val usesRs1 = id.valid && id.bits.rs1 =/= 0.U
val usesRs2 = id.valid && id.bits.rs2 =/= 0.U
val exeWrites = exe.valid && exe.bits.ctrl.regWrite && exe.bits.rd =/= 0.U
val loadUseOrAluHazard =
  exeWrites && ((usesRs1 && id.bits.rs1 === exe.bits.rd) ||
                (usesRs2 && id.bits.rs2 === exe.bits.rd))

when(loadUseOrAluHazard) {
  pcEnable := false.B       // Fetch 不前進
  idEnable := false.B       // Decode 保留同一條指令
  exeNext  := MicroOp.bubble
}
```

第二個修法是 forwarding：如果上一級已經算出 ALU result，就不要等 RF 寫回，直接把結果 bypass 到 operand。
Forwarding 的設計要明確列出來源優先序；越新的結果優先。

```text
RF value ─────────────┐
WB result ───────┐    │
EX result ──┐    │    │
            ▼    ▼    ▼
          priority mux ──► operand
          EX > WB > RF
```

```scala
def forward(rs: UInt, rfValue: UInt): UInt = {
  val fromExe = exe.valid && exe.bits.ctrl.regWrite && exe.bits.rd === rs && rs =/= 0.U
  val fromWb  = wb.valid  && wb.bits.rdWen            && wb.bits.rd  === rs && rs =/= 0.U
  Mux(fromExe, exeResult,
    Mux(fromWb, wb.bits.rdData, rfValue))
}

val op1 = forward(de.rs1, rf(de.rs1))
val op2 = forward(de.rs2, rf(de.rs2))
```

Load-use hazard 通常仍需要 stall，因為 load data 要等 memory response 才出來；ALU-to-ALU hazard 則可 forwarding。
完成後，同一段程式的 commit trace 應變成：

```text
cycle  pc    inst             rd  rdData
  3    0x00  addi x1,x0,1     x1  1
  4    0x04  add  x2,x1,x1    x2  2
  5    0x08  add  x3,x2,x1    x3  3
```

這個 walkthrough 的重點不是 forwarding 寫法本身，而是流程：先用最小程式製造錯誤，再用 trace 指出錯誤，
最後用 stall/forwarding 修正並讓同一個測試通過。CPU 教學中每加一個 feature 都應該維持這個節奏。

對應：§15。

## M6.3 微架構資料結構（朝 in-order+ / OoO 鋪路）

當 pipeline 變長，最重要的不是多加幾條 wire，而是定義穩定的 microarchitectural payload。
`MicroOp` 是 instruction 在 core 內部移動時的身分證：它攜帶 pc、inst、decode control、register id、
exception metadata 與 speculation metadata。只要這個結構清楚，bypass、scoreboard、replay、commit trace
都能圍繞它建立。

### `MicroOp` — pipeline 核心結構
```scala
class MicroOp extends Bundle {
  val valid = Bool()
  val pc, inst, imm = UInt(32.W)
  val ctrl = new DecodeSignals
  val rs1, rs2, rd  = UInt(5.W)
}
```
> 保留 `inst` 對 debug / trace / difftest 很有用。bubble 用 `MicroOp.bubble`（見 M3.5），不要塞 NOP。

### Decoder 輸出包成 Bundle（別散落 wire）
```scala
class DecodeSignals extends Bundle {
  val regWrite, memRead, memWrite = Bool()
  val aluOp  = UInt(4.W)
  val op1Sel, op2Sel = UInt(2.W)
}
```

### Exception / interrupt 用 `Valid[Payload]`
```scala
class ExceptionInfo extends Bundle { val cause = UInt(5.W); val tval, pc = UInt(32.W) }
val exception = Valid(new ExceptionInfo)     // ✗ 不要只用一個 Bool
```

### Memory response 用 tag（支援亂序回應）
```scala
class MemReq  extends Bundle { val addr = UInt(32.W); val tag = UInt(tagBits.W) }
class MemResp extends Bundle { val data = UInt(32.W); val tag = UInt(tagBits.W) }
```
用於：OoO memory response、MSHR、multicycle EU、TLB、BP query。

對應：§14–§15 延伸。

## M6.4 Flush 管理：epoch / branch mask

Flush 是 pipeline 最容易寫亂的控制之一。小 core 可以用全域 flush signal；設計變大後，
epoch 或 branch mask 能把「這筆資料是否還屬於目前控制流」放進 payload 本身。
這比到處廣播 kill signal 更可擴展，也更容易處理多個 outstanding branch 或 replay。

### Epoch（sequence number）殺 stale packet
```scala
val epoch = RegInit(false.B)
when(branchMispredict){ epoch := !epoch }
class FetchPacket extends Bundle { val pc, inst = UInt(32.W); val epoch = Bool() }
val killed = packet.epoch =/= currentEpoch    // 比全域廣播 flush 乾淨
```

### Branch mask（speculation 追蹤）
```scala
val branchMask = UInt(nBranches.W)
uop.branchMask := uop.branchMask & ~resolvedBranchOH         // resolve 清掉該 branch
val killed     = (uop.branchMask & mispredBranchOH).orR      // mispredict kill
```

## M6.5 Architectural vs Microarchitectural state（difftest 基礎）

差分測試只應比較架構狀態，因為不同實作可以有完全不同的微架構。ROB index、branch mask、
scoreboard busy bit 都是實作細節；pc、rd、rdData、trap 才是軟體可觀察的結果。
這個分界若不清楚，difftest 會變成綁死某一種 pipeline，而不是驗證 ISA 行為。
```scala
class ArchCommit  extends Bundle { val pc = UInt(32.W); val rd = UInt(5.W); val rdData = UInt(32.W); val rdWen = Bool() }
class MicroCommit extends Bundle { val robIdx = UInt(6.W); val branchMask = UInt(8.W); val exception = Bool() }
```
> **Differential testing 只看 architectural state**（pc/rd/rdData/trap），不看 ROB index 等 micro 細節。Commit trace 介面見 Part 8。

## M6.6 Performance counters

Performance counter 是理解微架構的最低成本儀表。`cycle` 和 `instret` 可以算 CPI；
stall breakdown 可以告訴你瓶頸在 frontend、backend、load-use、branch 還是 cache miss。
教學上不需要一開始就做完整 CSR，但應該讓學生學會用 counter 驗證「我加的 optimization 是否真的有效」。
```scala
val cycle   = RegInit(0.U(64.W)); cycle := cycle + 1.U
val instret = RegInit(0.U(64.W)); when(io.commit.fire){ instret := instret + 1.U }

class CorePerf extends Bundle {
  val cycle, instret = UInt(64.W)
  val frontendStall, backendStall, loadUseStall, branchFlush, cacheMiss = UInt(64.W)
}
```
教學上可觀察：CPI、branch penalty、cache miss penalty、stall breakdown。
