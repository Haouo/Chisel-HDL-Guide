# Part 2 — 建構區塊食譜（含 one-hot / 控制原語）

對應：書 §5–§7 ｜ `import chisel3._`、`import chisel3.util._`

> 你已懂這些區塊的原理；本章給「Chisel 標準寫法」與微架構控制神器（`Mux1H`/`PopCount`/`PriorityEncoderOH`/`BitPat`）。

本章把 Part 1 的 wire/reg/interface 組成可重用的小積木。對已學過數位邏輯的人來說，
ALU、counter、shift register、decoder 都不是新概念；真正要學的是 Chisel 中哪些工具能讓這些積木
在擴大後仍然清楚。例如 one-hot 選擇、priority encoder、population count 會在 cache hit、
issue queue、free list、bypass network 中反覆出現。

讀這章時，不要只問「這段 code 功能對不對」，還要問「它長成的邏輯深度如何」、
「有沒有 invariant 可以 assert」、以及「之後參數變大時是否仍可維護」。

**本章路線圖**

- 先學：ALU、Mux family、one-hot、counter/timer、memory 概覽、`BitPat` decoder。
- 會踩坑：把 priority/one-hot/lookup 語意混用，或忘記為 one-hot 假設加 assertion。
- 最後能做：把常見數位邏輯積木寫成可測、可擴、可 review 的 Chisel 標準寫法。

本章不是在背 utility API，而是在替常見控制語意選對硬體形狀：

| 問題 | 優先確認 |
|------|----------|
| 多路選擇 | 是 key lookup、priority，還是 one-hot |
| one-hot | 是否真的保證最多一個 bit，有沒有 assertion |
| counter/timer | wrap 發生在哪一拍，enable 停住時狀態如何 |
| decoder | illegal input 的 default control 是否明確 |

## M2.1 組合建構區塊

組合積木通常是資料路徑的葉節點。它們應該容易測、容易替換，而且不要偷偷引入狀態。
以 ALU 為例，最重要的是把 operation encoding、default output、以及非法 op 的行為定清楚；
否則 decoder 擴充後，很容易讓某些分支輸出未定或沿用舊值。

```scala
// 標記：概念示意。這些是常見資料路徑葉節點。
io.oh   := UIntToOH(io.in)          // binary → one-hot（或 1.U << io.in）
io.bin  := OHToUInt(io.in)          // one-hot → binary
io.prio := PriorityEncoder(io.in)   // 取最低位的 1 的 index
io.gt   := io.a > io.b              // comparator
```

### ALU（switch 選功能）
```scala
// 標記：教學骨架。可跑版見 ../examples/src/main/scala/handout/Combinational.scala 的 Alu。
class Alu extends Module {
  val io = IO(new Bundle { val a, b = Input(UInt(16.W)); val op = Input(UInt(2.W)); val y = Output(UInt(16.W)) })
  io.y := 0.U
  switch(io.op) {
    is(0.U){ io.y := io.a + io.b }; is(1.U){ io.y := io.a - io.b }
    is(2.U){ io.y := io.a & io.b }; is(3.U){ io.y := io.a | io.b }
  }
}
```

對應：§4.3、§5。

## M2.2 多路選擇：`MuxLookup` / `MuxCase` / `Mux1H`

多路選擇器不只是一個語法選擇，而是在描述不同種類的控制語意。`MuxLookup` 適合 key-value lookup；
`MuxCase` 表示有優先序的條件；`Mux1H` 則表示你相信 selector 最多只有一個 bit 有效。
這個語意差異很重要，因為它決定了你應該加什麼 assertion，也會影響生成邏輯的形狀。

```scala
// 標記：概念示意。三種 mux 對應不同控制語意。
// 小型選擇（curried 形式）
io.out := MuxLookup(io.op, 0.U)(Seq(
  AluOp.ADD -> (io.a + io.b),
  AluOp.SUB -> (io.a - io.b),
  AluOp.AND -> (io.a & io.b),
))

// 條件優先（先成立先選）
io.out := MuxCase(0.U, Seq(c0 -> v0, c1 -> v1))

// one-hot 選擇（sel 保證 one-hot 時最有效）
io.out := Mux1H(hitVec, dataVec)
```

| 場景 | 用什麼 |
|------|--------|
| 小型 key→value | `MuxLookup` |
| 條件優先序 | `MuxCase` / `PriorityMux` |
| one-hot select | `Mux1H` |
| FSM | `switch`/`is` |
| ISA decoder | decode table / `BitPat`（見 M2.6） |
| 複雜 control | `WireDefault` + local override |

## M2.3 One-hot 操作群（微架構控制核心）

One-hot 是微架構控制裡最常見的表示法之一。它讓「哪一個 entry 命中」、「哪一個 slot 被分配」、
「哪一路 bypass 被選到」這些問題可以用 bit mask 處理。只要使用 one-hot，就應該同時寫下 invariant：
例如 `PopCount(hitVec) <= 1.U`。這不是多餘的檢查，而是把設計假設變成可驗證的硬體契約。

```scala
// 標記：概念示意。重點是 one-hot invariant 要和 Mux1H 一起出現。
// cache hit way 選擇
val hitVec = VecInit.tabulate(nWays) { w => meta(w).valid && meta(w).tag === reqTag }
val rdata  = Mux1H(hitVec, dataVec)
assert(PopCount(hitVec) <= 1.U)            // ★ one-hot invariant

// allocation：找第一個 free
val freeOH = PriorityEncoderOH(~validMask)  // one-hot
val slot   = OHToUInt(freeOH)

// occupancy 統計
val numValid = PopCount(valids)
```

| 函數 | 用途 |
|------|------|
| `Mux1H(sel, seq)` | one-hot mux（issue select / bypass / hit way / grant） |
| `PriorityEncoderOH(v)` | 取最低 set bit 的 one-hot（free list / issue / RS allocation） |
| `OHToUInt` / `UIntToOH` | one-hot ↔ binary |
| `PopCount(v)` | 計數（ROB/LSQ/IQ/MSHR occupancy、branch mask count） |

### `UInt` bitmask vs `Vec[Bool]`
```scala
val maskUInt = valids.asUInt     // Vec[Bool] → UInt（做 bit op / PopCount / one-hot）
val maskVec  = maskUInt.asBools  // UInt → Vec[Bool]（做 per-entry index）
```
| 需求 | 選擇 |
|------|------|
| per-entry index | `Vec[Bool]` |
| bit operation / `PopCount` / one-hot | `UInt` mask |

## M2.4 時序建構區塊

時序積木的核心問題是「事件何時發生」以及「狀態何時更新」。Counter、timer、PWM、shift register
都看似簡單，但它們會訓練你處理 wrap、enable、initial value、以及 off-by-one。這些小錯誤若出現在
UART baud tick、cache refill counter 或 pipeline replay counter 中，會變得很難除錯。

```scala
val (value, wrap) = Counter(true.B, 10)             // counter

// timer / one-shot
val timer = RegInit(0.U(16.W)); val run = RegInit(false.B)
when(io.start){ timer := io.dur; run := true.B }
.elsewhen(run && timer =/= 0.U){ timer := timer - 1.U }
io.done := run && timer === 0.U

// PWM
class Pwm(period: Int) extends Module {
  val io = IO(new Bundle { val duty = Input(UInt()); val out = Output(Bool()) })
  val (cnt, _) = Counter(true.B, period); io.out := cnt < io.duty
}

// shift register（serial in / parallel out）
val shift = RegInit(0.U(8.W)); shift := Cat(shift(6,0), io.sin); io.pout := shift

val lfsr = chisel3.util.random.LFSR(16)             // 偽隨機
```

⚠️ **陷阱**：shift register 用 `Cat` 拼接，注意高/低位順序與寬度，否則位移方向相反。

對應：§6.2–6.3。

## M2.5 記憶體（概覽，深入見 Part 7）

記憶體不是大型 register array 的同義詞。不同 memory primitive 的讀取時序、資源映射與
read-under-write 行為都不同。教學上可以先用 `VecInit` 或小型 `Mem`，但只要目標接近 FPGA/ASIC，
就要開始把 `SyncReadMem` 的一拍延遲視為正常情況，並訓練 metadata 對齊。

```scala
val mem = SyncReadMem(depth, UInt(width.W))   // 同步讀（下一拍才有資料），對應 FPGA BRAM
io.rdata := mem.read(io.addr, io.re)
when(io.we){ mem.write(io.addr, io.wdata) }

val rom = VecInit(Seq(0.U, 1.U, 4.U, 9.U))    // ROM
io.out := rom(io.addr)
```

| 種類 | 寫法 | 讀取時序 |
|------|------|----------|
| 組合讀 RAM | `Mem(n, T)` | 同拍（小容量） |
| 同步讀 RAM | `SyncReadMem(n, T)` | **下一拍**（FPGA BRAM） |
| ROM | `VecInit(Seq(...))` | 組合 |

⚠️ **陷阱**：`SyncReadMem` 的 read-during-write 行為未定義；需 forwarding（Part 7 詳述）。

對應：§6.4。

## M2.6 ISA decoder：`BitPat` / `ListLookup`

**一句話**：用位元樣式描述指令編碼，讓 decoder 貼近 ISA encoding table。

Decoder 是「硬體控制」與「規格表」交會的地方。`BitPat` 的好處是讓程式碼看起來接近 ISA encoding table：
固定 bit 寫出來，不關心的 bit 用 `?`。這比一堆手寫 slice 比較更容易 review，也更容易新增指令。
大型 decoder 仍然應該把輸出 control 包成 Bundle，避免控制線散落到整個 core。

```scala
val ADD  = BitPat("b0000000??????????000?????0110011")
val SUB  = BitPat("b0100000??????????000?????0110011")
val ADDI = BitPat("b?????????????????000?????0010011")

// inst 命中哪個樣式 → 取對應 control 欄位
val ctrl = ListLookup(inst, defaultCtrl, Array(
  ADD  -> addCtrl,
  SUB  -> subCtrl,
  ADDI -> addiCtrl,
))
```

> `?` 表示 don't care bit。大型 decoder 可用 `chisel3.util.experimental.decode.decoder`（邏輯最小化）。

對應：§14（processor decode）。

## M2.7 輸入處理

外部輸入是硬體設計中最容易被教學範例忽略的地方。按鈕、GPIO、跨時脈訊號都不是乾淨同步訊號；
直接送進主邏輯可能在模擬中看起來正常，實際硬體卻因 metastability 或 bouncing 出現偶發錯誤。
因此 synchronizer、edge detector、debounce/filter 應該被視為標準邊界電路。

```scala
val sync   = RegNext(RegNext(io.asyncIn))     // 2-FF synchronizer（跨時脈域必備）
val prev   = RegNext(io.in); io.rising := io.in && !prev   // edge detect
val samples = RegInit(0.U(3.W)); samples := Cat(samples(1,0), io.in)
io.filtered := PopCount(samples) >= 2.U       // 3 取 2 多數決
```

⚠️ **陷阱**：**任何**外部非同步輸入先過 2-FF synchronizer，否則 metastability。

對應：§7。

**練習**
- ★ 用 `Mux1H`+`PopCount` 寫 4-way cache hit 選擇並加 one-hot assertion。完成標準：故意製造雙命中時 assertion 會失敗。
- ★ 用 `BitPat` 寫 RV32I 的 R-type decoder（ADD/SUB/AND/OR）。完成標準：非法指令回到明確 default control。
- ★★ 用 `PriorityEncoderOH` 寫 8-entry free-list allocator。完成標準：分配、釋放、全滿三種情況都有測試。
