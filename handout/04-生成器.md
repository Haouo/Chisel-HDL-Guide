# Part 4 — 硬體生成器（parameterized RTL generator）

對應：書 §10 ｜ `import chisel3._`、`import chisel3.util._`

> Chisel 的價值不是少寫 Verilog，而是 **以可維護、可參數化的方式描述一整個硬體設計家族**。
> 隨時自問：這段 Scala elaboration 後變成什麼硬體？

本章開始把 Chisel 當成 generator language。前幾章教的是如何寫出一份乾淨 RTL；
這章教的是如何寫出一整族乾淨 RTL。參數化的難點不是把 `width` 換成變數，而是讓非法組合早點失敗、
讓 derived parameter 集中、讓 optional hardware 的存在與否清楚可追，並且讓不同設計點能用同一套測試驗證。

好的 generator 應該有三個特徵：生成期資料流清楚、生成出的硬體容易預測、參數變動時錯誤能在 elaboration
就被抓到。若 generator 需要讀者在腦中追一堆隱式副作用，它只是把 Verilog 的問題搬到 Scala。

**本章路線圖**

- 先學：Scala 生成硬體、`case class Params`、`Option`/ADT、generic module、`reduceTree`、sweep。
- 會踩坑：參數散落、非法組合太晚爆、用 `reduce` 生成長 critical path、top IO 寬度推斷。
- 最後能做：把單一 RTL 擴展成一族可參數化、可命名、可 sweep 的 generator。

## M4.1 用 Scala 生硬體

用 Scala 生硬體時，function call 不是 runtime 呼叫硬體，而是在 elaboration-time 展開結構。
`delay(io.in, 3)` 會長出三個 register，不是產生一個會執行三次的硬體函數。這個觀念決定了
你能不能正確使用遞迴、`for`、`map` 與 helper function。

```scala
// function 回傳硬體（elaboration-time generator，不是 runtime call）
def delay(x: UInt, n: Int): UInt = if (n == 0) x else delay(RegNext(x), n - 1)
io.out := delay(io.in, 3)                          // 生成 3 級 register

// immediate generator（純組合 helper，最安全）
def immI(inst: UInt): UInt = Cat(Fill(20, inst(31)), inst(31, 20))

// 用 for 生成重複結構
val regs = Seq.fill(4)(RegInit(0.U(8.W)))
for (i <- 0 until 3) { regs(i + 1) := regs(i) }     // 串成 pipeline
```

純組合 helper 適合：immediate gen、bit slicing、mask 生成、branch target、ALU helper。
⚠️ **陷阱**：在 function 裡偷改 register 會讓資料流變隱形；helper 盡量保持純組合。`println`/`if`/`for` 都在生成期跑。

對應：§10.1–10.3。

## M4.2 用 `case class Params` 集中參數

**一句話**：把參數打包，並在裡面算好 derived parameter、用 `require` 守住非法組合。

當參數超過兩三個，直接傳 `width: Int, depth: Int, ...` 會讓 constructor 變得難讀，也讓 derived
parameter 散落各處。`case class Params` 應該成為設計契約：哪些組合合法、哪些欄位由使用者指定、
哪些欄位由其他參數推導。這樣 sweep design space 時，錯誤會在生成期爆出來，而不是在模擬很久後才發現。

```scala
case class CacheParams(addrBits: Int, nSets: Int, nWays: Int, blockBytes: Int) {
  require(isPow2(nSets),      "nSets must be power of two")
  require(isPow2(blockBytes), "blockBytes must be power of two")
  // derived parameter 集中在這，不要散落各處
  val offsetBits = log2Ceil(blockBytes)
  val indexBits  = log2Ceil(nSets)
  val tagBits    = addrBits - indexBits - offsetBits
}

class Cache(p: CacheParams) extends Module {
  val tag = UInt(p.tagBits.W)        // 直接用 derived param
}
```

| 好處 | 說明 |
|------|------|
| 集中管理 | 參數與 derived 一處 |
| 早期失敗 | 非法組合在 elaboration 就爆 |
| 易 sweep | design-space exploration 方便 |

⚠️ **陷阱**：不要到處重算 `log2Ceil(p.nRobEntries)`；放進 params 算成 `robIdxBits`，用 `UInt(p.robIdxBits.W)`。

對應：§10.4。

## M4.3 `Option` 表 optional hardware；ADT 表 generator choice

Optional hardware 的核心是「不存在時不要長出硬體，也不要留下半接的 wire」。`Option` 很適合描述有無；
sealed trait ADT 則適合描述互斥選擇。相比一堆 Boolean，ADT 可以避免 `useBtb=false` 但
`useGshare=true` 這種語意衝突，也能把各選項自己的參數放在對應 case class 中。

### Optional hardware
```scala
val btbOpt = if (p.useBtb) Some(Module(new BTB(p))) else None
for (btb <- btbOpt) { btb.io.update := upd }        // 存在時才連
// optional port
val debug = if (p.hasDebug) Some(Output(UInt(32.W))) else None
debug.foreach { d => d := 0xDEAD.U }
```
適合：optional cache / predictor / debug port / perf counter / mul-div。

### 用 Scala ADT（sealed trait）表選擇，而非一堆 Boolean
```scala
// ✗ 一堆 bool，組合爆炸又難約束
case class CoreParams(useBtb: Boolean, useBht: Boolean, useGshare: Boolean)

// ✓ ADT：互斥選擇清楚、攜帶各自參數
sealed trait BpKind
case object NoBP                                  extends BpKind
case class  Bimodal(entries: Int)                extends BpKind
case class  GShare(entries: Int, historyBits: Int) extends BpKind

p.bp match {
  case NoBP                  => /* 無 predictor */
  case Bimodal(n)            => val bp = Module(new BimodalPredictor(n))
  case GShare(n, h)          => val bp = Module(new GSharePredictor(n, h))
}
```

## M4.4 Generic module：傳 `gen: T`、用 `chiselTypeOf`

Generic module 的用途是把 protocol 或結構抽象出來，而不是抽象掉硬體語意。例如 queue、register slice、
skid buffer 可以對任意 payload 型別工作，因為它們只關心 valid-ready 與 storage，不關心 payload 欄位。
但 ALU、decoder 這類依賴資料語意的模組，不應該為了泛型而犧牲可讀性。

```scala
class RegSlice[T <: Data](gen: T) extends Module {
  val io = IO(new Bundle { val in = Flipped(Decoupled(gen)); val out = Decoupled(gen) })
  val bits = Reg(chiselTypeOf(io.in.bits))     // 由值取型別
}

// generic hardware helper
def holdUnless[T <: Data](x: T, en: Bool): T = {
  val r = Reg(chiselTypeOf(x)); when(en){ r := x }; r
}
```

| 工具 | 用途 |
|------|------|
| `[T <: Data](gen: T)` | 任意 Data 型別參數 |
| `chiselTypeOf(x)` / `cloneType` | 由硬體值取得型別樣板 |

常用於：generic queue / pipe / skid buffer / SRAM wrapper。
⚠️ **陷阱**：type parameter 一定要傳「型別樣板」（如 `UInt(8.W)`），否則 Chisel 不知寬度。

## M4.5 函數式生成（map / reduce / **reduceTree**）

函數式集合操作會直接影響硬體形狀。`map` 生成一批平行邏輯；`reduce` 常生成線性鏈；
`reduceTree` 生成平衡樹。這不是單純的 Scala style，而是 timing tradeoff。當輸入數量變大，
線性鏈可能成為 critical path，因此設計 generator 時要把硬體拓撲納入 API 選擇。

```scala
// ★ reduceTree 是 Vec 的方法（不是 Seq）；Seq 要先 VecInit。
//   用 +& (width-expanding add) 避免加總溢位。
val inputs = VecInit(io.a, io.b, io.c, io.d)
io.sum := inputs.reduceTree(_ +& _)                      // 平衡樹，O(log n) 延遲
io.min := inputs.reduceTree((x, y) => Mux(x < y, x, y))  // minimum search

// VecInit.tabulate 生規律硬體
val hits = VecInit.tabulate(nWays){ w => ways(w).valid && ways(w).tag === reqTag }

// arbitration tree：元素是 (Bool, UInt) tuple（非 Data），不能用 Vec.reduceTree，
// 改用自寫的泛型平衡樹歸約 helper：
def treeReduce[T](xs: Seq[T])(op: (T, T) => T): T =
  if (xs.length == 1) xs.head
  else { val (l, r) = xs.splitAt((xs.length + 1) / 2); op(treeReduce(l)(op), treeReduce(r)(op)) }

def arbitrate(reqs: Seq[(Bool, UInt)]): (Bool, UInt) =
  treeReduce(reqs) { (l, r) => (l._1 || r._1, Mux(l._1, l._2, r._2)) }
```

| 集合函數 | 生成的硬體 |
|----------|-----------|
| `.map(f)` | 每元素一份硬體 |
| `.reduce(_ op _)` | **線性鏈**（O(n) 延遲） |
| `Vec.reduceTree(_ op _)` | **平衡樹**（O(log n)，timing 佳）— 限 `Vec`/`Data` |
| 自寫 `treeReduce` | 平衡樹，適用任意 Scala 型別（tuple…） |
| `VecInit.tabulate(n)(f)` | 規律 Vec（cache way / RF） |

⚠️ **陷阱**
- `reduce` 生線性長鏈（critical path 差）；要短延遲用 `reduceTree`。
- `reduceTree` **只在 `Vec` 上**（`Seq` 要 `VecInit` 包起來）；非 `Data` 型別用自寫 `treeReduce`。
- module 的 **top-level port 不可用 inferred width**（`Output(UInt())` 會被 firtool 拒絕）；對外 port 給明確 `.W`。

對應：§10.6。

## M4.6 module 命名與 design-space sweep

當你一次產生很多設計點，命名就是除錯工具。`desiredName` 會進到 Verilog module name、波形階層、
報告與產物目錄；如果所有版本都叫 `Core`，後續比較 timing、area、測試失敗會很痛苦。
好的名稱應該包含影響硬體形狀的關鍵參數，而不是把整個 config dump 進去。

```scala
class Core(p: CoreParams) extends Module {
  override def desiredName = s"Core_${p.issueWidth}w_${p.nRobEntries}rob"  // 影響 Verilog/波形/report
}

// 一次產生多組 config
val configs = Seq(
  CoreParams(issueWidth = 1, nRobEntries = 16),
  CoreParams(issueWidth = 2, nRobEntries = 32),
  CoreParams(issueWidth = 4, nRobEntries = 64),
)
configs.foreach { p =>
  circt.stage.ChiselStage.emitSystemVerilog(new Core(p),
    firtoolOpts = Array("-o", s"generated/${p.name}"))
}
```
可 sweep：cache size、associativity、queue depth、issue width、predictor entries、MSHR count。

## M4.7 繼承與重用

Scala 繼承可以重用結構，但在 Chisel 中要小心：硬體 ownership 與 initialization order 比一般軟體更敏感。
如果只是共用 IO 或少量 helper，abstract base class 可以接受；若行為差異主要由參數決定，
通常 `case class Params`、ADT 與組合式 helper 會比深繼承更容易追。

```scala
abstract class Processor extends Module {
  val io = IO(new Bundle { val pc = Output(UInt(32.W)); val halt = Input(Bool()) })
}
class SimpleCpu extends Processor { /* 共用上面 io */ }
```
⚠️ **陷阱**：abstract 宣告的 `io` 子類別沿用；別在子類別重複 `IO(...)` 同名介面。

對應：§10.5。

**練習**
- ★ 把 ALU 改成 type-parameterized（吃 `UInt` 或 `SInt`）。完成標準：測試能證明兩種型別都可 elaborate。
- ★ 用 ADT 寫 branch predictor 選擇（NoBP/Bimodal/GShare），各自 instantiate。完成標準：非法組合無法被表示成參數。
- ★★ 用 `case class CacheParams` + `require` + `desiredName` 寫可 sweep 的 cache 骨架，產生 3 種 config 的 Verilog。完成標準：非法 `nSets` 會在 elaboration-time 失敗。
