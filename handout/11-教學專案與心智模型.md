# Part 11 — 教學專案實作建議（Chipyard-lite）與最終心智模型

對應：reference 教學篇 + 心智模型 ｜ 把前 10 章收斂成可落地的專案結構

最後一章的目標是把前面所有技術收斂成一個可教、可做、可長大的專案形狀。
完整 Chipyard 很強，但它把太多大型系統問題一次帶進來：implicit parameters、Diplomacy、binder、
device tree、harness、software flow。對教學 CPU 或個人 SoC，直接搬全套通常會讓學生先迷失在框架裡，
反而看不到 core、bus、device、harness 的基本邊界。

Chipyard-lite 的策略是借精神，不借重量。保留 config-driven design、core/SoC/harness 分離、
MMIO address map、commit trace、simulation collateral 分層；暫時不用 full Diplomacy 與 implicit parameter graph。
這樣學生先建立 local reasoning，日後讀 Rocket/BOOM 時才知道那些大型機制是在解哪個已經熟悉的問題。

**本章路線圖**

- 先學：為什麼不直接搬 full Chipyard、Chipyard-lite 目錄、simple bus、core/device/harness 分層。
- 會踩坑：core 寫死 peripheral address、simulation-only logic 混進 RTL、config 只變成散落 Boolean。
- 最後能做：把 Wildcat 重組成可分階段驗收的教學 SoC 專案。

本章的重點不是新增更多功能，而是把責任切乾淨。每個 milestone 都要通過三個門檻：

| 門檻 | 問題 |
|------|------|
| 邊界 | 新功能應該放 core、soc、devices、harness，還是 config |
| 契約 | 它和其他層透過哪個正式介面溝通 |
| 驗收 | 關閉該 config 或替換 harness 時，target RTL 是否仍乾淨 |

## M11.1 不建議直接搬 full Chipyard

教學專案最重要的是讓錯誤可定位。若一開始就引入 full Chipyard，學生寫錯一個 peripheral address decode，
可能要追 Config fragment、LazyModule graph、TL node、binder、harness bridge 才找到問題。
這對初學者不是必要複雜度。先用明確 case class、手寫簡單 bus、清楚 module hierarchy，能把硬體本質看清楚。

對教學 CPU / 個人 SoC 專案，**不要一開始**引入：
```text
full Diplomacy / site·here·up / LazyModule everywhere
heavy implicit Parameters / Rocket-style cake pattern
```
原因：local reasoning 變差、學生看不到硬體、debug 成本太高、焦點偏離 CPU/SoC 本質。

## M11.2 做 Chipyard-lite：借精神、去重量

目錄分層是架構決策，不只是整理檔案。`core/` 不應依賴 device；`soc/` 負責把 core、memory、bus、
device 接起來；`harness/` 負責模擬環境；`config/` 描述要組哪一種設計。這種結構讓每個 bug
先落在合理範圍內：ISA bug 看 core，address decode bug 看 soc，UART console bug 看 harness/device 邊界。

```text
src/
  core/      Core.scala  CommitTrace.scala
  soc/       SoCTop.scala  SimpleBus.scala  AddressMap.scala  Devices.scala
  devices/   UART.scala  Timer.scala  Framebuffer.scala  Halt.scala
  harness/   TestHarness.scala  SimUART.scala  SimFramebuffer.scala  ISSCosim.scala
  config/    Config.scala  SmallCoreConfig.scala  PipelineCoreConfig.scala  CacheConfig.scala
```

### Mini config（用 case class，先別用 implicit Parameters）
```scala
// 標記：概念示意。這是教學版 config data，不使用 Rocket implicit Parameters。
case class SoCConfig(core: CoreConfig, memory: MemoryConfig,
                     devices: Seq[DeviceConfig], harness: HarnessConfig)
```
> 比一開始用 Rocket `implicit Parameters` 更適合教學：參數來源明顯、可 grep、可 local reason。

### 用 ADT 表 device，不用一串 Boolean

```scala
// 標記：概念示意。ADT 表示互斥/可列舉的 device 選項，避免散落 Boolean。
sealed trait DeviceConfig { def name: String; def base: BigInt; def size: BigInt }
case class TimerConfig(base: BigInt = 0x10010000L, size: BigInt = 0x1000) extends DeviceConfig {
  val name = "timer"
}
case class UARTConfig(base: BigInt = 0x10011000L, size: BigInt = 0x1000, baudDiv: Int = 868) extends DeviceConfig {
  val name = "uart"
}
case class HaltConfig(base: BigInt = 0x10012000L, size: BigInt = 0x1000) extends DeviceConfig {
  val name = "halt"
}

case class HarnessConfig(enableTrace: Boolean = true, randomMemLatency: Boolean = false, timeoutCycles: Int = 10000)
case class SmallSoCConfig() {
  val soc = SoCConfig(
    core = CoreConfig(rv32 = true),
    memory = MemoryConfig(base = 0x80000000L, size = 64 * 1024),
    devices = Seq(TimerConfig(), UARTConfig(), HaltConfig()),
    harness = HarnessConfig()
  )
}
```

這種寫法保留了 Chipyard config-driven 的精神：改 variant 時改 config，不改 `Core` 或 `SoCTop` 主體。
同時它比 implicit `Parameters` 更直白，學生可以用 grep 找到每個 device 從哪裡來。

## M11.3 Simple bus / address map（讓學生看見 MMIO decode）

手寫 simple bus 的價值在於透明。學生可以直接看到 address range 如何命中 device、讀寫如何產生 side effect、
read data valid 如何延遲對齊。等這些都理解後，再使用 TileLink、AXI 或 Wishbone 才有意義；
否則 protocol library 會變成黑箱。

```scala
// 標記：概念示意。Simple bus 的教學重點是 address range ownership。
case class AddressRegion(name: String, base: BigInt, size: BigInt)

def hit(addr: UInt, r: AddressRegion): Bool =
  addr >= r.base.U && addr < (r.base + r.size).U
```
讓學生理解：address map、MMIO decode、device selection、bus routing。

### Simple bus 也要有明確 protocol

即使教學 bus 很簡單，也應該說清楚 request/response timing。否則一接 `SyncReadMem`、MMIO side effect
或 random memory latency，bug 會變得很難追。

```scala
// 標記：教學骨架。真正實作時需補 arbitration、latency、error 與 device side effect policy。
class BusReq extends Bundle {
  val addr = UInt(32.W)
  val wdata = UInt(32.W)
  val wmask = UInt(4.W)
  val isWrite = Bool()
}

class BusResp extends Bundle {
  val rdata = UInt(32.W)
  val error = Bool()
}

class SimpleBusPort extends Bundle {
  val req = Flipped(Decoupled(new BusReq))
  val resp = Decoupled(new BusResp)
}
```

建議契約：

| 項目 | 教學版建議 |
|------|------------|
| request | `req.fire` 表示地址與 write data 被接受 |
| response | read/write 都回一筆 `resp`，方便對齊 exception/error |
| latency | RAM 可固定一拍，device 可固定一拍，進階再加 random latency |
| decode miss | 回 `error=true`，不要靜默讀 0 |
| side effect | write side effect 在 `req.fire && isWrite` 發生 |

address map 集中在 `soc/AddressMap.scala`，device 不應自己決定全系統地址。device 只知道自己的 local register offset。

## M11.4 Core 不應知道 peripheral 細節（關鍵分層）

Core 的責任是執行 ISA 並發出 memory transaction，不是知道 UART base address 或 framebuffer layout。
Peripheral address 屬於 SoC integration；terminal/SDL/ISS 屬於 harness。這個分層一旦守住，
core 可以在不同 SoC 中重用，也可以用不同 harness 做測試。

| ✗ 不建議 | ✓ 建議 |
|----------|--------|
| Core 內部寫死 UART/framebuffer/halt 的 address | Core 只發 memory request |
| | Bus decoder 決定去 RAM / UART / framebuffer / halt |
| | Peripheral 各自實作 side effect |

## M11.5 Harness 分離 simulation-only logic

Harness 不是可有可無的 testbench，而是設計可攜性的保護層。模擬時要載入程式、印 UART、
接 framebuffer、跑 ISS cosim、設 timeout；FPGA/ASIC 時這些東西都會換掉。把它們留在 harness，
可以讓 target RTL 保持乾淨。

| Core / SoC RTL（真硬體） | TestHarness（sim 環境） |
|--------------------------|-------------------------|
| core、bus、device 邏輯 | ELF loader、SDL framebuffer、terminal UART、ISS cosim、timeout、random memory latency |

> 這很 Chipyard，但輕量、適合教學——也讓設計能無痛搬上 FPGA/ASIC。

### Verification config（用 config 開關，不混進硬體產品）
```text
WithISSLockstep / WithWaveform / WithCommitTrace
WithPerfCounters / WithRandomMemoryLatency / WithUARTConsole
```

### Harness 只碰正式邊界

Harness 可以載入 ELF、接 UART console、跑 golden model，但它應該只透過正式介面碰 target RTL：
memory port、UART pins/MMIO bridge、commit trace、interrupt line、reset/clock。不要讓 harness 伸手讀 core
裡的任意 register；需要觀測內部時，用 Part 9 的 Probe/Layer 明確標記成 verification-only。

```text
TestHarness
  ├─ ProgramLoader  → memory init / bus model
  ├─ SimUART        → UART TX/RX interface
  ├─ GoldenModel    → commit trace
  ├─ Timeout        → simulation stop condition
  └─ Wave/Trace     → debug artifact

SoCTop
  ├─ Core
  ├─ SimpleBus
  ├─ RAM
  └─ MMIO devices
```

這個分界對日後上 FPGA 很關鍵：`SoCTop` 可以留下，`TestHarness` 全部換成 board shell。

## M11.6 最終心智模型

整份講義最後要建立的不是某個 API 清單，而是一個設計判斷：
Chisel 讓你用 Scala 組織硬體，但不會替你取消 RTL discipline。每個 abstraction 都要能回答
「生成什麼硬體、時序在哪裡、契約如何驗證、產物如何整合」。只要這四個問題答得出來，
從小模組到 SoC generator 都是同一套思維。

### Chisel 進階技巧 = 「如何優雅地產生 RTL」
```text
Bundle / Vec / Decoupled / Queue
SyncReadMem / BlackBox
case class Params / require / Option / ADT
assert / cover / perf counters
trace / difftest / harness
```

### Chipyard 進階技巧 = 「如何讓眾多 generator/protocol/sim/軟體 metadata/FPGA·ASIC flow 被同一套 config 組裝」
```text
Config fragments / implicit Parameters / LazyModule / Diplomacy
TileLink nodes·widgets / RegMapper / IOBinders·HarnessBinders
Bridge pattern / generated metadata / FireSim·Hammer integration
```

### 最重要的一句話
> **Chisel 不是讓你逃離 RTL discipline，而是讓你在保留 RTL discipline 的前提下，
> 建立可組合、可參數化、可驗證、可延展的硬體生成系統。**

## M11.7 學習路線收尾

```text
基礎能力（Part 0–6）：寫得出乾淨、可測的 RTL，理解最小 CPU/pipeline 的核心路徑與 hazard 修法
進階能力（Part 7–9）：會接記憶體/時脈/黑盒、建驗證基礎建設、用 CIRCT 新特性
架構視野（Part 10–11）：讀得懂 Rocket/BOOM，能設計可組裝的 SoC generator
```

真正的目標不是「Can I write less Verilog?」，而是
**「Can I express a family of microarchitectures without losing RTL discipline?」**

### Capstone milestones：從 Wildcat 到 Chipyard-lite

把 capstone 拆成小里程碑，比一次要求「做一個 SoC」更可控。每個 milestone 都應該能獨立跑測試，
並且只引入一種新的架構邊界。

每個 milestone 都採同一個交付格式：

```text
1. 新增哪個邊界
2. 哪些 package dependency 被允許
3. 至少一個會先失敗的測試
4. 關閉或替換該功能時，哪些產物不應再出現
```

| Milestone | 新增內容 | 檔案重點 | 驗收方式 |
|-----------|----------|----------|----------|
| 1. Core boundary | 把 Wildcat 包成 `Core`，只保留 instruction/data memory 介面與 commit trace | `core/Core.scala`, `core/CommitTrace.scala` | core 不 import `devices`/`harness`；ALU 程式能 commit 正確 trace |
| 2. Simple bus | 新增 `SimpleBus`、`AddressMap`，把 RAM 與 MMIO decode 分開 | `soc/SimpleBus.scala`, `soc/AddressMap.scala` | 對 RAM 與非法地址各有測試；read data valid 對齊 |
| 3. Timer device | 加 MMIO `Timer`，含 counter、compare、interrupt/status bit | `devices/Timer.scala` | 軟體寫 compare 後，counter 到達時 status/interrupt 變 true |
| 4. UART device + harness | 加 MMIO UART TX register，harness 接 `SimUART` 收字元 | `devices/UART.scala`, `harness/SimUART.scala` | bare-metal 程式寫 UART，測試端收到預期字串 |
| 5. Config variants | 用 case class 描述 Small / Pipeline / WithTimer / WithUART | `config/Config.scala` | 切 config 不改 core/soc 主體；不存在的 device 不產生 decode |
| 6. Difftest mode | `WithCommitTrace` 打開 golden model 對拍，關閉時不產生 cosim | `harness/ISSCosim.scala` | 相依指令測試能抓 forwarding bug；關閉 config 後 core RTL 無 cosim 依賴 |

建議的最小檔案形狀：

```text
src/main/scala/
  core/
    Core.scala
    Decode.scala
    CommitTrace.scala
  soc/
    SoCTop.scala
    SimpleBus.scala
    AddressMap.scala
    Ram.scala
  devices/
    Timer.scala
    UART.scala
    Halt.scala
  harness/
    TestHarness.scala
    SimUART.scala
    GoldenModel.scala
  config/
    Config.scala
```

每個 milestone 的測試都應該先寫失敗案例。例如 Timer 先測 compare 還沒到時 interrupt 不該起來；
UART 先測 back-to-back write 不應覆蓋 busy byte；Difftest 先用 Part 6 的 data hazard 程式讓錯誤 pipeline 失敗。
這樣 capstone 不是單純堆功能，而是在建立可維護的 SoC 開發節奏。

## M11.8 Chipyard-lite 的實作順序

不要一次把 core、bus、device、harness、config 全部重寫。每一步只引入一個新邊界，並且保留可跑測試。

1. 先把 Wildcat 包成 `Core`：輸入 instruction/data memory 介面，輸出 `CommitTrace`。
2. 加 `Ram` wrapper：固定 read latency，測 metadata 對齊。
3. 加 `SimpleBus`：只有 RAM 與 decode miss，先不加 device。
4. 加第一個 MMIO `Halt` device：用最小 side effect 驗證 address decode。
5. 加 `Timer`：驗證 register map、counter、compare、interrupt/status。
6. 加 `UART` 與 `SimUART`：第一次碰 harness/target 分離。
7. 加 `SoCConfig` variants：讓不存在的 device 不生成 decode。
8. 加 difftest：commit trace 對 Scala golden model。

每一步的完成標準都要能用一句話判斷，例如「core 不 import `devices`」、「decode miss 會回 error」、
「關閉 UART config 後 generated RTL 沒有 UART module」。這些標準比「程式可以跑」更能防止架構慢慢混在一起。

### Package dependency 應該長這樣

```text
core      → chisel/util only
devices   → bus local register interface
soc       → core + devices + address map + memory
harness   → soc + sim models + golden model
config    → pure Scala config values
```

不應該長這樣：

```text
core imports UARTConfig
core imports SimUART
device imports TestHarness
```

若 dependency 開始反向，通常代表某個 address、simulation behavior 或 verification concern 放錯層了。

**綜合練習（capstone）**
- ★★ 用 Chipyard-lite 結構重組你的 Wildcat：core / soc / devices / harness / config 分層。完成標準：core 不 import device 或 harness package。
- ★★★ 加一個 MMIO `Timer` 與 `UART`，bus decoder 路由；harness 接 SimUART 印字。完成標準：address map 集中定義，device side effect 有測試。
- ★★★ 加 `WithCommitTrace` config，commit trace 對拍一個 Scala golden model（Part 8）。完成標準：關閉 config 時不產生 trace/harness cosim 邏輯。
