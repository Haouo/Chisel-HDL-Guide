# Part 8 — 驗證與除錯基礎建設（assert / cover / difftest / harness）

對應：書 §13 ｜ `import chisel3._`、`import chisel3.util._`

> 大型 Chisel 設計的關鍵不是「能跑」，而是「能持續驗證」。逐步累積自己的 verification vocabulary。

本章把測試從「寫幾個 expect」提升到「建立可長期維護的驗證基礎建設」。硬體設計的錯誤常常不是單一輸入下
算錯，而是 protocol 假設被打破、pipeline 邊界條件沒定義、memory latency 沒對齊、或 harness logic
混進可合成 RTL。這些問題需要 assertion、coverage、trace、difftest 與清楚分層一起處理。

學完本章後，讀者應該能為自己的設計列出 invariants，知道哪些 assertion 應該貼在 RTL 旁邊，
哪些測試應該放在 harness，commit trace 應該包含哪些 architectural state，以及如何避免 debug 訊號把正式 RTL 弄髒。

**本章路線圖**

- 先學：assert/cover、reusable checker、printf/println、測試分層、commit trace、harness 分離。
- 會踩坑：只測 happy path、trace 提早在 execute 階段產生、把 ELF loader/timeout 放進 core。
- 最後能做：用 checker 與 golden model 把 protocol 與 architectural state 固定成可回歸測試。

---

## M8.1 `assert` / `cover` 是大型設計必備

Assertion 是把設計假設寫成硬體檢查。它不只是防 bug，也是在告訴未來的讀者：這裡期待 one-hot、
這裡不允許 full 還 enqueue、這裡 x0 永遠為 0。Coverage 則回答另一個問題：重要情境到底有沒有被測到。
沒有 cover，測試通過可能只是因為 branch miss、cache miss、stall、flush 從來沒發生。

```scala
// protocol invariants
assert(!(io.enq.fire && full),  "enq when full")
assert(!(io.deq.fire && empty), "deq when empty")
// cache invariant
assert(PopCount(hitVec) <= 1.U, "multiple hit ways")
// scoreboard invariant
when(io.wb.valid && io.wb.bits.rd =/= 0.U){ assert(busy(io.wb.bits.rd)) }

// coverage：確認重要事件有被測到
cover(io.branchMispredict)
cover(io.cacheMiss)
cover(io.loadUseHazard)
cover(io.storeLoadForward)
```

## M8.2 建立 reusable checkers（累積 verification vocabulary）

**一句話**：別到處寫零散 assertion；包成可重用 checker object。

當同一種 protocol 出現在很多地方，checker 應該被抽成 vocabulary。這會讓設計風格逐漸收斂：
所有 ready-valid 都有 stalled stability 檢查，所有 cache hit vector 都有 one-hot 檢查，
所有 register file 都檢查 x0。大型專案的品質往往來自這些小 checker 的長期累積。

```scala
object CacheChecker {
  def oneHotHit(hitVec: Seq[Bool]): Unit = assert(PopCount(VecInit(hitVec)) <= 1.U)
}
object PipelineChecker {
  def x0AlwaysZero(x0: UInt): Unit = assert(x0 === 0.U, "x0 must be zero")
}
// ready-valid 穩定性 checker（見 Part 3 M3.3）
RVAssert.stableWhenStalled(io.mem.req)
```

## M8.3 除錯訊號：`printf` / `println` / `Printable`

`println` 與 `printf` 對應兩個不同世界。`println` 在 elaboration-time 執行，適合印參數與生成決策；
`printf` 會成為模擬中的硬體輸出，適合印每拍 signal。混淆這兩者會導致「為什麼我的 signal 印不出來」
或「為什麼這個 log 只出現一次」這類錯誤。

```scala
println(s"Generating cache with ${p.nWays} ways")            // elaboration-time（生成 log）
printf(p"pc=${Hexadecimal(pc)} inst=${Hexadecimal(inst)} rd=${rd}\n")  // simulation runtime
```
> `p"..."`（`Printable`）可內插 Chisel 值並指定格式（`Hexadecimal`/`Decimal`/`Binary`），是 signal formatting 首選。

## M8.4 `dontTouch` 不是除錯第一選擇

`dontTouch` 很方便，但它是鈍器。短期 debug 可以用它防止訊號被最佳化，但長期應該改成正式 debug IO、
Probe、Layer 或 trace 介面。否則 generated RTL 會充滿臨時訊號，影響最佳化與後端報告，也讓設計邊界變得不清楚。

```scala
dontTouch(debugSignal)        // 臨時：阻止被最佳化掉
```
| 建議優先序 | 手段 |
|-----------|------|
| 1 | 正式 debug IO |
| 2 | Probe / Layer（見 Part 9） |
| 3 | `dontTouch`（臨時） |

⚠️ **陷阱**：太多 `dontTouch` 會讓 generated RTL 變髒、影響最佳化。

## M8.5 測試分層（unit → sim → cosim）

不同層級的測試解不同問題。Unit test 應該快而精準，抓 decoder、ALU、FIFO 這種局部 bug；
Chisel simulation 測 protocol 與多 cycle 行為；full-system simulation 測軟體可觀察結果；
difftest 則用 golden model 鎖住 ISA 行為。不要用 full-system 測試取代 unit test，否則失敗時定位成本會很高。

| 層 | 工具 | 測什麼 |
|----|------|--------|
| 1 | Scala / ChiselSim unit test | decoder、immgen、ALU、FIFO |
| 2 | Chisel simulation | pipeline handshake、cache FSM、branch predictor |
| 3 | Verilator / C++/Python driver | full CPU、跑 ELF 程式、長時測試 |
| 4 | **ISS differential testing** | RTL commit trace vs ISS golden model |

## M8.6 Commit trace 與 differential testing

**一句話**：每拍 commit 一條 architectural state，與 ISS 對拍比對。

Commit trace 是 CPU 驗證的關鍵邊界。它應該在指令完成、架構狀態即將被觀察時產生，而不是在 decode
或 execute 階段提早猜測。Trace 欄位要少而穩定：pc、inst、rd/rdData、trap、memory side effect 等
軟體可觀察狀態。這讓 RTL pipeline 可以改，difftest 介面仍然維持。

```scala
class CommitTrace extends Bundle {
  val valid  = Bool()
  val pc, inst, rdData = UInt(32.W)
  val rdWen  = Bool(); val rd = UInt(5.W); val trap = Bool()
}
```
```
RTL commit one instruction  ─┐
ISS step one instruction    ─┴─► compare pc / rd / rdData / trap
```
> 只比 **architectural** state（Part 6 M6.5）；ROB index 等 micro 細節不比。

### Walkthrough：最小 Scala golden model 對拍

在教學 CPU 中，不必一開始就接完整 ISS。可以先寫一個只支援少數指令的 Scala golden model，
讓它和 RTL commit trace 一拍一拍比對。這個版本支援 `addi`、`add`、`and` 已足夠抓 data hazard、
x0 寫入、pc 更新與 writeback 錯誤。

第一步，RTL 每次真正提交一條指令時送出 trace：

```scala
class CommitTrace extends Bundle {
  val valid  = Bool()
  val pc     = UInt(32.W)
  val inst   = UInt(32.W)
  val rd     = UInt(5.W)
  val rdWen  = Bool()
  val rdData = UInt(32.W)
  val trap   = Bool()
}

io.commit.valid       := wb.valid
io.commit.bits.pc     := wb.bits.pc
io.commit.bits.inst   := wb.bits.inst
io.commit.bits.rd     := wb.bits.rd
io.commit.bits.rdWen  := wb.bits.ctrl.regWrite && wb.bits.rd =/= 0.U
io.commit.bits.rdData := wbResult
io.commit.bits.trap   := false.B
```

第二步，在 Scala 測試端維護一份 golden architectural state：

```scala
final case class GoldenState(pc: BigInt = 0, regs: Vector[BigInt] = Vector.fill(32)(0)) {
  def step(inst: BigInt): (GoldenState, Option[(Int, BigInt)]) = {
    val opcode = inst & 0x7f
    val rd     = ((inst >> 7)  & 0x1f).toInt
    val funct3 = (inst >> 12) & 0x7
    val rs1    = ((inst >> 15) & 0x1f).toInt
    val rs2    = ((inst >> 20) & 0x1f).toInt
    val funct7 = (inst >> 25) & 0x7f
    val immI   = BigInt(inst.toInt >> 20) & 0xffffffffL

    val result =
      if (opcode == 0x13 && funct3 == 0) regs(rs1) + immI       // addi
      else if (opcode == 0x33 && funct3 == 0 && funct7 == 0) regs(rs1) + regs(rs2) // add
      else if (opcode == 0x33 && funct3 == 7) regs(rs1) & regs(rs2)               // and
      else throw new IllegalArgumentException(f"unsupported inst 0x$inst%x")

    val nextRegs = if (rd == 0) regs else regs.updated(rd, result & 0xffffffffL)
    (copy(pc = pc + 4, regs = nextRegs), if (rd == 0) None else Some(rd -> (result & 0xffffffffL)))
  }
}
```

第三步，在 ChiselSim 中每看到一筆 commit，就讓 golden model step 一次並比較：

```scala
var golden = GoldenState()

if (dut.io.commit.valid.peek().litToBoolean) {
  val pc   = dut.io.commit.bits.pc.peek().litValue
  val inst = dut.io.commit.bits.inst.peek().litValue
  assert(pc == golden.pc, s"pc mismatch rtl=$pc golden=${golden.pc}")

  val (next, writeback) = golden.step(inst)
  writeback.foreach { case (rd, data) =>
    dut.io.commit.bits.rd.expect(rd.U)
    dut.io.commit.bits.rdData.expect(data.U)
  }
  golden = next
}
```

如果 Part 6 的 forwarding 還沒做好，這個測試會在 `add x2,x1,x1` commit 時失敗。這比只看最後 `dbg`
輸出更好，因為錯誤被定位在「哪一條指令、哪個 rd、哪個 cycle」。

實務上 golden model 會逐步長大：先支援 ALU subset，再加 load/store，再加 branch/trap。
每加一類指令，都先新增會失敗的 trace test，再修改 RTL。這就是 CPU 專案的 red-green-refactor。

## M8.7 Simulation-only logic 放 harness，不要塞進 core

Core 與 harness 的分離是可攜性的底線。ELF loader、timeout、ISS cosim、隨機 memory latency、
terminal UART 這些東西對模擬很有用，但不是產品硬體。它們應該圍在 SoC 外面，透過正式 memory/bus/trace
介面互動。這樣同一份 core 才能搬到 FPGA、ASIC 或不同模擬環境。

| 應放 TestHarness / sim wrapper | 不要放 core RTL |
|-------------------------------|-----------------|
| ELF loader、SDL framebuffer、terminal UART | — |
| ISS cosim、timeout checker、random memory latency model | — |

> core/SoC 是真硬體；harness 是 target environment。混在一起會讓 RTL 無法合成、難移植（FPGA/ASIC）。

## M8.8 Full-system regression（CPU 專案）

Full-system regression 的價值在長期保護行為。當 core 開始支援 branch、load/store、interrupt、cache，
單元測試無法涵蓋所有交互作用；這時需要一組可重跑的 bare-metal 程式、ISA test、stress test 與 timeout。
每次改 pipeline 或 memory system，都應該重跑這些測試，確認沒有破壞軟體可觀察語意。

建議測：
```
rv32ui / rv32mi subset
自寫 bare-metal C tests
cache stress / branch stress
interrupt / timer tests
Snake game smoke test（互動 + framebuffer + UART）
```

## M8.9 從設計契約推導驗證計畫

進階驗證不是把測試數量堆高，而是把設計契約逐條固定下來。每個 module 都應該能回答三件事：
合法輸入下保證什麼、非法輸入如何被阻擋、哪些重要情境必須被 cover。這個思路會一路延伸到 Part 10
的 TileLink/RegMapper：protocol contract 比單筆 expect 更重要。

### 例：FIFO 的驗證切法

| 契約 | 驗證手段 |
|------|----------|
| empty 時不能 deq.fire | RTL assertion |
| full 時不能 enq.fire | RTL assertion |
| stalled 時 bits 保持穩定 | reusable ready-valid checker |
| enqueue 順序等於 dequeue 順序 | scoreboard / reference queue |
| full、empty、simultaneous enq/deq 都發生過 | cover |

```scala
val model = scala.collection.mutable.Queue[BigInt]()

if (dut.io.enq.fire.peek().litToBoolean) {
  model.enqueue(dut.io.enq.bits.peek().litValue)
}
if (dut.io.deq.fire.peek().litToBoolean) {
  val expected = model.dequeue()
  dut.io.deq.bits.expect(expected.U)
}
```

### CPU/SoC 專案的驗證階梯

| 階段 | 要固定的契約 | 失敗時看哪裡 |
|------|--------------|--------------|
| Unit | decoder、ALU、FIFO、immediate、branch target | 單一 module |
| Protocol | ready-valid、memory latency、MMIO side effect | module 邊界 |
| Pipeline | forwarding、stall、flush、commit order | core microarchitecture |
| Difftest | pc、rd、trap、memory side effect | architectural state |
| Full-system | bare-metal program、interrupt、cache stress | SoC/harness 互動 |

常見錯誤是直接從 unit test 跳到 full-system regression。這樣測試看起來很完整，但一旦失敗，
定位範圍太大。比較好的節奏是：每新增一個架構邊界，就加一層對應的 checker 或 reference model。

### Assertion 放哪裡

| Assertion 類型 | 建議位置 |
|----------------|----------|
| module 自己永遠應該滿足的 invariant | RTL 旁邊 |
| protocol 使用者必須遵守的假設 | interface checker |
| 只在某個 test/harness 有意義的條件 | harness / test |
| 昂貴 shadow state 或 coverage | Layer（Part 9） |

> 原則：越接近設計契約、越應該靠近 RTL；越接近測試策略、越應該放 harness。

**練習**
- ★ 為你的 FIFO 加 `assert(!(enq.fire && full))` 與 `cover(full)`，跑 ChiselSim。完成標準：故意改壞 full 判斷時 assertion 能抓到。
- ★★ 為 Wildcat 拉出 `CommitTrace`，寫一個簡易「golden model（Scala）」對拍比較。完成標準：trace 只含 architectural state。
- ★★★ 把 ELF loader 與 timeout 移到 TestHarness，core 只留 memory 介面。完成標準：core RTL 不含檔案路徑、plusarg 或模擬專用 class。
